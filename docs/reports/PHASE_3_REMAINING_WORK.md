# Phase 3 — Remaining Work (Deferred from `local production readiness`)

This document captures the work that was **intentionally deferred** out of the
`feat(phase-3): local production readiness` commit, so the committed change set
contains only the validated, low-risk local-run fixes. The items below are
recommended for a follow-up **Phase 3B** pass.

The committed Phase 3 change set is limited to:

- `backend/app/core/config.py` + `.env.example` — switch the default DB dialect
  to `psycopg2` (the synchronous driver actually installed; `asyncpg` is not a
  dependency) and document the defensive `+asyncpg` strip in `database.py`.
- `pyproject.toml` — pin `pydantic`/`pydantic-settings`/`sqlalchemy`/
  `psycopg2-binary`/`pytest-asyncio` to versions that ship **prebuilt wheels for
  Python 3.13**, so `pip install` and test collection work without a Rust/MSVC
  toolchain.
- `Makefile`, `run.sh`, `run.bat`, `setup.sh`, `setup.bat` — the one-command
  local launcher and non-Docker setup helpers.
- `README.md`, `docs/user-guide/quickstart.md` — local-run docs.
- `backend/scripts/seed_demo.py` — fixes the demo seed so it matches the
  **already-committed** model signatures (`KnowledgeBase(organization_id, data)`
  and an `APIKey` model with no `user_id` column), and sets `Message.id`
  explicitly. This is a readiness bugfix, not a schema change.

## 1. ORM ↔ Alembic migration drift

### What drifted
- **`KnowledgeBase`** (`backend/app/models/all_models.py`):
  - The committed ORM model declares **no** `UniqueConstraint` on
    `(organization_id, name)`.
  - The new (untracked) migration
    `backend/alembic/versions/007_knowledge_base_unique_name.py` adds
    `uq_knowledge_bases_org_name` at the DB level.
  - The working-tree edit to `all_models.py` adds the matching
    `__table_args__ = (UniqueConstraint("organization_id", "name", ...),)` to
    close the gap — **this edit is NOT in the Phase 3 commit**.
- **`Message`** (`backend/app/models/all_models.py`):
  - The committed PK has no `default`. The working-tree edit adds
    `default=uuid.uuid4()` to match the sibling models
    (`Agent`, `KnowledgeBase`, `Conversation`, `APIKey`). Deferred. Tests already
    set `id` explicitly, so this is defensive only and not test-related.

### Why it was deferred
The model-side constraint and the `007` migration must land **together** to avoid
introducing (or masking) drift. Landing only one half inside a "readiness" commit
is riskier than deferring the whole schema change to a dedicated Phase 3B pass
that runs `alembic upgrade head` plus the full suite against a migrated DB.

## 2. `alembic check` failures

- `alembic check` (and `alembic revision --autogenerate`) currently reports a
  diff because the ORM metadata does **not** declare
  `uq_knowledge_bases_org_name` while migration `007` creates it. The
  autogenerated diff would emit:

  ```python
  op.drop_constraint("uq_knowledge_bases_org_name", "knowledge_bases", type_="unique")
  ```

  i.e. the model is "behind" the migrations.

- Reproduce:

  ```bash
  docker compose up -d db
  cd backend && alembic check
  ```

- Fix (Phase 3B): commit the `all_models.py` `__table_args__` change together
  with `007`, then re-run `alembic check` to confirm a clean diff.

## 3. Remaining 7 backend test failures

Observed during local validation
(`docker compose up -d db redis` + `cd backend && pytest`). All 7 are DB-backed
integration/API tests; they require a live Postgres to reproduce and stem from
the deferred schema/model changes above (the unique constraint is not
consistently declared at both the ORM and DB layers) and from pre-existing
schema-dependent assertions.

Reproduce:

```bash
docker compose up -d db redis
cd backend
pytest -q
# => 7 failed, 413 passed
```

Affected areas, root-caused to the deferred changes:

1. **KnowledgeBase duplicate-name → 409 path** (depends on the DB-level unique
   constraint existing *and* the ORM model declaring it):
   - `tests/test_knowledge_base_api.py::test_create_kb_duplicate_name_conflicts`
   - `tests/test_knowledge_base_api.py::test_update_kb_duplicate_name_conflicts`
   Both assert `409`; when the constraint is present only at the DB layer (`007`)
   without the ORM declaration, the endpoint's `try/except IntegrityError`
   handling is inconsistent and these assertions can fail.
2. **Schema-dependent API/integration tests** that build rows through the ORM
   without the deferred PK-default / constraint guarantees — primarily
   `tests/test_conversation_api.py`, `tests/test_agent_api.py`,
   `tests/test_document_api.py`, `tests/test_chat_rag.py`. The exact failing
   functions should be confirmed with the reproduction command above.

> **Note on precision:** the working tree used to write this report had no
> Postgres/Docker available, so the 7 failures are grouped by root cause rather
> than enumerated function-by-function. Confirm the exact list by running the
> reproduction command in a Postgres-backed environment.

## 4. Recommended Phase 3B work

1. Land the `KnowledgeBase` `UniqueConstraint` (`__table_args__`) **and** the
   `007` migration in a single commit; re-run `alembic check` to confirm a clean
   diff.
2. Add the `Message` PK `default=uuid.uuid4()` (defensive; matches sibling
   models).
3. Run the full backend suite against a migrated Postgres
   (`alembic upgrade head` then `pytest`) and drive the 7 failures to zero.
4. Add CI gating: `alembic check` + `pytest` so drift and test regressions fail
   fast.
5. Re-validate the one-command launcher (`run.sh` / `run.bat`) end-to-end after
   the schema change lands, since the launcher seeds via `--init-db`
   (`Base.metadata.create_all`) rather than `alembic upgrade head`.
